# Workflow 工作流引擎

> **以工具驱动执行为核心，并通过 `计划 -> 执行 -> 审查` 循环保障任务质量和完成可靠性。**

ChatSpeed 的 Workflow (`src-tauri/src/workflow/react`) 是一个**一等公民的任务执行引擎**——它并不仅仅是聊天会话的扩展。它的底层基础是 **工具驱动的 ReAct 运行时**：任务推进应当通过工具调用、结构化观察、审批和显式完成来收敛，而不是停留在自由对话本身。

从产品视角看，整个工作流围绕 **`计划 -> 执行 -> 审查`** 主循环构建：

- **计划**：明确接下来怎么做，减少盲目执行
- **执行**：通过具体工具动作真正推动任务前进
- **审查**：检查质量、发现回归，并形成清晰的收口边界

正是这条循环，让它特别适合编程、调试、重构以及其他那些“质量依赖真实执行，而不是依赖会说”的任务。

---

## 🎯 核心架构

Workflow 工作流引擎由以下几个关键运行时模块协同工作：

| 组件 | 作用 |
|------|------|
| **WorkflowExecutor** | 主 ReAct 循环执行器——管理 AI 对话、工具调用、状态转换和生命周期 |
| **WorkflowManager** | 全局会话注册表——管理多个并发工作流会话 |
| **子任务编排模块** | 通过 child task 与 orchestrator 相关模块派生隔离的子任务执行单元 |
| **IntelligenceManager** | AI 驱动的摘要、质量审计和工具审批审查 |
| **ContextManager** | 负责构建与重建对话历史和模型侧上下文投影 |
| **ContextCompressor** | 智能分级上下文压缩，减少 Token 用量 |
| **MemoryManager** | 跨会话持久化记忆（项目级和全局） |
| **PathGuard** | 文件系统访问控制，阻止敏感系统路径 |
| **SkillScanner** | 发现和集成可复用的技能包 |
| **LoopDetector** | 滑动窗口循环检测，防止无限执行 |

## 🚀 核心功能

### 🔧 工具驱动执行

工作流不是简单的"聊天"，而是通过以下步骤推进：

1. **规划** — AI 创建结构化的执行计划
2. **工具调用** — AI 调用工具（文件编辑、网络抓取、Shell 等）
3. **观察** — 系统返回结构化的工具执行结果
4. **审批** — 潜在风险的工具调用等待用户确认
5. **完成提交** — 通过 `complete_workflow_with_summary` 显式完成任务

在这个模型里，工具调用不是附属能力，而是任务推进的主路径。允许有简短推理，但只要任务仍未完成，工作流就应尽快收敛到下一个具体工具动作。

### 🔁 `计划 -> 执行 -> 审查` 循环

工作流的质量保障来自一条可重复的主循环：

1. **计划** —— 生成或修正下一步执行策略
2. **执行** —— 通过工具完成当前最小但最有价值的动作
3. **审查** —— 检查结果、审批信息、文件差异、审计结论，再决定下一步或最终收口

这条循环带来的直接价值包括：

- 减少只有分析没有推进的空转
- 让执行过程通过结构化工具结果变得可观察
- 在任务完成前形成显式审查边界
- 让长链路编程任务更不容易漂移失控

### 🧭 运行时不变量

工作流运行时遵循几条关键架构规则：

- **后端状态是绝对权威** —— 会话存活、运行时状态、等待原因、审批状态和可恢复性都由后端掌控。
- **结构化状态优先于转录文本** —— 恢复和 UI 同步依赖结构化事件、工具元数据和执行上下文，而不是重新解析助手文本。
- **`messages` 是持久化历史** —— 对话转录历史是权威来源。
- **`context_messages` 是可重建的上下文投影** —— 它服务于模型上下文效率，而不是成为第二份真相来源。
- **等待态是显式建模的** —— 通过 `WorkflowState` 和 `WaitReason` 表达，而不是从消息文本中推断。

### ♻️ 稳定状态恢复

恢复不是“尽量而为”的体验补丁，而是运行时能力的一部分。

- 审批态、等待态、执行态都由后端显式建模
- 工作流被中断后，可以基于结构化状态恢复，而不是依赖重新解析聊天文本
- 如果用户在计划审批或工具审批期间关闭软件，下次进入时仍可继续当前审批和后续运行，而不是被迫从头开始
- 已完成工作流在合法追问场景下也支持受控热恢复

### 🧩 规划/执行模型分离

工作流支持**为规划和执行使用不同的模型**：
- **规划模型**：用于策略制定的高能力模型（如 `deepseek-v4`、`claude-4-sonnet`）
- **执行模型**：用于具体执行的高性价比模型（如 `gemma3-12b`、`qwen3-code`）

这样可以在保证规划质量的同时降低成本。

### 🧵 真正的后台多工作流执行

这不是“单个聊天线程伪装成工作流”。

- 用户可以同时运行多个工作流
- 每个工作流都有独立的运行时状态、审批链路、上下文和进度跟踪
- 用户可以在多个活跃工作流之间自由切换，而不会把它们压扁成一个线程
- 子智能体和后台执行模式可以让长任务持续推进，用户则可以去处理别的事情

### 🧵 子智能体编排

工作流可以**派生子智能体**来并行处理子任务：
- `execution_mode="call"` — 主工作流暂停，等待子智能体完成
- `execution_mode="background"` — 子智能体在后台并行运行，主智能体继续执行

### ✅ 审批与审查系统

- **自动审批列表**：信任的工具自动执行，无需确认
- **智能 AI 审查**：AI 对未知工具生成风险评估
- **完全人工审批**：每个工具调用都需用户确认
- **最终审计模式**：完成前可选的独立代码审查环节
- **Shell Policy**：Shell 命令审批的自定义规则

审批与审查结合在一起后，工作流就不再是“AI 一路尝试直到它觉得自己做完了”，而更像一条带有可见质量关卡的受控执行链路。

### 🔔 全局通知

即使用户当前没有盯着某个工作流窗口，后台执行也不会“悄悄发生然后悄悄结束”。

- 后台工作流出现待审批事项时，可以触发全局通知
- 任务完成时也可以触发完成通知
- 通知形式包括软件状态提示和可选声音播放，让用户不必持续盯着界面，也能及时发现审批和完成事件

### 🧠 上下文压缩

引擎使用**智能分级压缩**，按任务边界进行压缩：

| 模式 | 行为 |
|------|------|
| `Rollup` | 非阻塞压缩，执行过程中流式进行 |
| `Blocking` | 完全暂停执行，直到压缩完成 |

压缩规则：
- 保留最近完成的任务
- 只压缩较早的已完成工作
- 初始压缩在完成 3 个任务且有新活动任务时触发
- 深度集成 = 长任务续航能力

> 以 `deepseek-v4` 的实际测试为例，上下文输入缓存命中率可达到约 **96%**。

### 💾 记忆系统

两个作用域的持久化记忆：

| 作用域 | 路径 | 用途 |
|--------|------|------|
| **全局记忆** | `~/.chatspeed/memory.md` | 跨项目 AI 知识 |
| **项目记忆** | `~/.chatspeed/project/{transformed_path}/memory.md` | 项目特定上下文 |

记忆文件保存在项目目录外，避免被 Git 追踪，也方便在多个工作流会话之间复用。

### 🔒 安全体系

工作流引擎提供多层安全保障：

- **PathGuard**：阻止访问敏感系统路径（`/etc`、`/bin`、`C:\Windows` 等）
- **工具白名单**：限制 AI 可以调用的工具
- **自动审批**：配置信任的工具静默执行
- **Shell Policy**：Shell 命令执行规则定义
- **授权路径**：配置授权的工作区目录

即便用户选择自动放行较多审批，系统级破坏性操作仍应被运行时安全策略拦截。工作流的执行边界被限制在授权目录内，而不是获得无限制的整机访问能力。

### 🛡️ 更低打扰但更安全的 Bash 执行

这个工作流不是靠“疯狂弹审批”来体现安全，也不是靠“完全放开 Shell”来追求顺滑。

- 对只读、读取型 Bash 操作可以更积极地自动放行
- Shell Policy 与路径校验共同限制执行边界
- 这样既能减少正常工程任务里的审批打断，又不会牺牲核心安全模型
- 换句话说，目标是在保证系统安全的前提下，让工作流执行过程中的审批尽可能少

### ⚙️ 状态机

工作流引擎通过完整的状态机运作：

| 状态 | 含义 |
|------|------|
| `Pending` | 会话已创建，尚未开始 |
| `Thinking` | AI 正在推理 |
| `Executing` | 正在执行工具 |
| `Auditing` | 最终审计中 |
| `Stopping` | 正在取消 |
| `Paused` | 等待继续/停止确认 |
| `AwaitingUser` | 等待用户文本输入 |
| `AwaitingApproval` | 等待工具审批 |
| `AwaitingAutoApproval` | AI 辅助审批审查中 |
| `AwaitingSubAgent` | 等待子智能体结果 |
| `Completed` | 任务成功完成 |
| `Error` | 不可恢复的错误 |
| `Cancelled` | 用户取消 |

所有等待相关状态转换都是显式的，恢复与继续执行会基于当前 `wait_reason` 做校验。

## 🔄 与其他模块的集成

### Skills（技能系统）
工作流可以从多个来源加载和使用**技能包**：
- 用户安装的技能 (`~/.chatspeed/skills`)
- 内置技能 (`@chatspeed/skills`)
- 工作流管理的技能配置

### MCP 工具
工作流直接集成 ChatSpeed 的 MCP 代理，允许 AI 调用：
- `WebSearch` / `WebFetch` — 互联网访问
- 已安装的 MCP 工具 — Context7、Tavily、Puppeteer 等

### AGENTS.md
工作流支持通过 `AGENTS.md` 文件进行项目级智能体配置，定义智能体角色、工具访问权限和行为配置。

## 🎬 快速开始

1. **创建新工作流** — 从 ChatSpeed 的工作流侧边栏创建
2. **选择模型** — 为规划和执行选择模型（或使用同一模型）
3. **配置工具** — 选择启用的技能和 MCP 工具
4. **开始任务** — 描述你需要完成的工作
5. **监控进度** — 查看流式输出、审批工具调用、审查结果
6. **完成审查** — 任务完成后查看最终审计报告（如启用）

### 工作流 UI 组件

ChatSpeed 提供专属的工作流界面：
- **工作流侧边栏** — 会话管理和多工作流切换
- **消息列表** — 流式对话输出与工具执行详情
- **审批弹窗** — 工具调用审批/拒绝界面
- **文件差异预览** — 审查改动的并排对比视图
- **任务账本** — 待办列表和工具执行追踪
- **状态面板** — 当前工作流状态和智能体进展
- **模型选择器** — 每个工作流独立的模型配置
- **技能选择器** — 每个工作流的技能包选择
